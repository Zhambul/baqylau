# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A kitty-terminal integration for Claude Code, built entirely out of Claude Code **hooks** plus detached background processes. Three user-facing features:

1. **Tab colors** (`claude-tab-status.py`) — the kitty tab color reflects the session state (grey idle · magenta busy · blue running/awaiting · red asking-you · green your-turn) via `kitten @ set-tab-color` over the socket in `$KITTY_LISTEN_ON`.
2. **Command mirror pane** (`claude-split.py` + `claude-mirror.py`) — a right-side vertical split showing every command, file op, subagent, teammate, monitor, and codex run as colored streaming blocks — fg/bg command blocks carry clickable ` ⧉cmd ⧉out` copy links, and Read/Update/Write one-liners click-to-EXPAND their content in place — highlighted code for Read/Write, a line-numbered soft-red/green diff for Update; second click collapses (both via OSC 8 hyperlinks → `claude-copy.py` via `~/.config/kitty/open-actions.conf`, docs/wiring.md / docs/click-to-view.md) — plus a 5-row scoreboard window (`claude-scorebar.py`) underneath (session id · ✉ message census · ▪ activity (cmds + time) · Σ token breakdown + cost · files + ± line-diff + tools).
3. **Audit trail** (`core/audit.py`, CLI `bin/claude-audit.py`) — always-on SQLite recording of every hook event, tab transition, slot claim, stream lifecycle, paint op, and swallowed exception, at `~/.claude/kitty-audit/audit.db`.
4. **Web dashboard** (`dashboard/` + `bin/claude-dashboard.py`, docs/dashboard.md) — a localhost web UI (127.0.0.1:8377, `CLAUDE_DASH_PORT` overrides) over EVERY session, live and parked: the mirror stream rendered to HTML (`dashboard/opshtml.py` — html-escape is the `neutralize()` analog; ⧉ copy / click-to-view work via the same `core/copy.collect` + kv stashes; MAIN-AGENT-ONLY, unlike the terminal mirror: agent/teammate/secondary-codex ops carry a producer-source stamp — `core/ops.py` `src`, ambient per-process `set_src`/`$CLAUDE_OPS_SRC` — and `op_items` drops them; agent detail lives in the drill-down), per-agent activity drill-down (`plugins.activity()` timelines), ctx-saturation bars (own row, account-limit-bar styling) on the session cards / session header / agent cards, git branch+worktree chips with a dirty `*` (`git_info` — .git-file reads for branch/worktree; the dirty flag is the ONE sanctioned `git status` subprocess, TTL-cached per cwd; a linked worktree also carries `root`, its owning main checkout — the list page groups sessions by `root||cwd`, so worktree sessions aggregate under their project, docs/dashboard.md *Grouping and titles*) (the path-keyed `plugins.context()` fan-out — `transcript.context_probe` bounded-tail probe over the last assistant usage, docs/dashboard.md *Context saturation*), cross-session toast/OS notifications when a tab goes red (asking you) or green (done) — a 1s `tab_states()` diff watcher over SSE — and a full control plane (message composer — on a PARKED session it becomes "resume & send": the message rides a `--resume` relaunch's argv through the existing web-launch path, docs/dashboard.md *Resume & send* — interrupt, mid-turn cancel-edit, close, rename, ⇆ migrate (the manual trigger of the relimit account migration — no confirm, no % ceiling, docs/relimit.md *Manual migrate*) — the ✎ header button appends the `agent-name` naming record through its single owner `transcript.set_session_title` (the ONE control-plane session-state write, docs/dashboard.md *Web rename*) + retitles a live tab via the new `Frontend.set_tab_title` (sticky), works live AND parked — quick commands — a second scoreboard action row typing the TUI's own `/compact` + `/model` + `/effort` (closed vocabulary, `post_command`, docs/dashboard.md *Web quick commands*), and FULL rewind: pick any prompt bubble + restore mode and `dashboard/rewindmenu.py` drives Claude Code's own checkpoint menu with screen-verified key events — docs/dashboard.md *Web rewind*; and FULL AskUserQuestion support: the ask card mirrors the TUI dialog — options/multiSelect/free-text/chat-about-this — via the `ask-pending` kv stash (`plugins/claude_code/ask_fmt.py`) and `dashboard/askdialog.py` driving the real dialog, docs/dashboard.md *Web ask*; and FULL plan-mode support: the plan card renders the pending ExitPlanMode plan + the dialog's own decision options fetched from the live screen, `dashboard/plandialog.py` — approve/feedback/dismiss, docs/dashboard.md *Web plan mode*; and a PINNED TASKS CARD at the top of the mirror tab — the session's native TaskCreate/TaskUpdate list, completed rows struck-through/dimmed, in_progress accented — fed by the `tasks` kv `task_fmt.py` re-snapshots from Claude Code's on-disk task dir on every task-touching hook (the dir is wiped at session end, so the kv is what parked sessions show; a status flip fires NO dedicated hook — its PostToolUse is the only refresh signal), docs/dashboard.md *Web tasks*; and MIC DICTATION on the composer + the new-session prompt — Deepgram Nova-3 streaming, live interim results spliced into the textarea; the server only mints a ~30s grant JWT (`dashboard/dictate.py`, `POST /api/dictate/token`, key at `~/.config/deepgram/api-key`, keyterms file alongside) and the browser talks wss to Deepgram DIRECTLY (the stdlib server can't speak WS and never sees audio), docs/dashboard.md *Web dictation*). A CONSUMER tier above core/plugins (imports both, nothing imports it back); read-only server (`ThreadingHTTPServer`, per-request `mode=ro`, singleton pid-lock on `paths.DASH_DB` + port bind, `stream_lifecycle` kind `dashboard`).

There is no build system or package manifest. Scripts are invoked directly by hooks wired in `~/.claude/settings.json` (the hook table is in docs/wiring.md — the settings file itself is *not* in this repo). Python scripts target the system `python3`; only `pygments` is an (optional, probed-for) runtime dependency. The e2e test suite lives in `tests/` (`make test`, dev-only deps in `requirements-dev.txt` — see docs/testing.md); it drives the real hook scripts as subprocesses against per-test temp dirs and a fake `kitten` recorder.

**Layout (core / plugins / frontends — docs/architecture.md).** `core/` is the tool- and terminal-agnostic runtime; `plugins/<tool>/` adapts one agent tool each (`claude_code` — a host; `codex` — a secondary source inside a Claude session AND a standalone host on its own, via codex's native SessionStart hook → `claude-codex-session.py`), registered in `plugins/__init__.py` (`all_plugins` / `on_session_start` / `census` / `activity` — the last is the drill-down timeline fan-out behind `core/sessionapi.py`, docs/sessionapi.md); both hosts drive the shared `core/hostpane.py` pane lifecycle; a third plugin `plugins/otel/` is NOT an agent tool but a cross-cutting subsystem — a per-machine singleton OpenTelemetry OTLP receiver (`claude-otlp-receiver.py`, spawned at SessionStart when `CLAUDE_CODE_ENABLE_TELEMETRY=1`) that is the AUTHORITATIVE source of the scoreboard's token/cost counters (it captures Claude Code's hidden `auxiliary` agents that transcript-folding never could; the transcript fold survives only as a SessionEnd fallback). Raw datapoints land in the audit `otel` table; `frontends/` adapts one terminal each behind the `Frontend` interface (`frontends.get()`; kitty is the only implementation, `$CLAUDE_FRONTEND` overrides). Dependency rule: core imports only core; frontends at most core; plugins import core + frontends, never each other; `dashboard/` (the web-dashboard consumer tier, docs/dashboard.md) sits above all — it imports core + the plugins registry root + frontends (the last for its control plane: the write endpoints reach the terminal through `frontends.get()`), and nothing imports it back except its bin/ entry and tests. The `bin/` directory holds EVERY executable entry point (`bin/claude-*.py` — hook entries, renderers, streamers, the audit CLI `bin/claude-audit.py`, plus `bin/retarget-python.py`); their FILENAMES are load-bearing (they name the audit's handler/script vocabulary) — never rename an entry (moving is fine, the vocabulary is the basename); put implementation in the packages, and spawn siblings via `core/paths.py`'s `BIN`, never a re-derived path. **Naming convention (deliberate, not drift):** hyphenated `claude-*.py` = executable entry, un-importable by design; underscored `*.py` inside `core/`/`plugins/`/`frontends/` = importable module. The repo root contains no Python files. **All hook events are wired (in `~/.claude/settings.json`, outside the repo) to ONE dispatcher entry, `claude-hook.py` → `plugins/claude_code/dispatch.py`** — except `WorktreeCreate`/`WorktreeRemove`, which must stay UNWIRED (delegating hooks: registering one overrides Claude Code's native worktree creation and must print the worktree path; the dispatcher's silent exit-0 broke `EnterWorktree` — docs/wiring.md), which reads the payload once and fans out IN-PROCESS to the tab dispatch, the matcher-selected formatter, and the universal audit subscriber (matcher routing lives in `dispatch.py`'s `_plan()`; docs/wiring.md). Because every subsystem runs under argv[0]=`claude-hook.py`, the dispatcher stamps `audit.set_handler("claude-<x>.py")` around each call so `hook_events.handler`/`errors.script` keep their per-entry vocabulary (argv[0] is no longer the sole source — an explicit override is). The per-script entry shims (`bin/claude-cmd-fmt.py` …) still exist and still run standalone (the e2e tests drive them directly). The historical root-level compat shims (`claude_state.py`, `claude_ops.py`, …) are DELETED — import the package modules directly (`core.state`, `core.ops`, `plugins.claude_code.hookkit`, …).

## Commands

```sh
# Audit CLI — the primary debugging tool (run from repo root)
python3 bin/claude-audit.py sessions            # recent sessions
python3 bin/claude-audit.py anomalies <sid>     # canned queries for known bug signatures — start here
python3 bin/claude-audit.py errors    <sid>     # swallowed exceptions, full tracebacks
python3 bin/claude-audit.py timeline  <sid>     # merged chronological story of a session
python3 bin/claude-audit.py otel      <sid>     # OTEL cost/token breakdown by query_source (main/subagent/auxiliary)
python3 bin/claude-audit.py sql "<query>"       # free-form read-only SQL (sql-write for fixups)

# Manual smoke test — cycle the tab colors (~3s each)
for s in idle thinking working executing awaiting-bg awaiting-command awaiting-response; do
  ./bin/claude-tab-status.py "$s"; ping -c 4 127.0.0.1 >/dev/null
done
./bin/claude-tab-status.py clear

# Mirror pane controls
./bin/claude-split.py toggle|grow|shrink|reset|setpct <N>

# Web dashboard (docs/dashboard.md)
./bin/claude-dashboard.py            # start if needed + open the browser
./bin/claude-dashboard.py serve|start|stop|status

# E2E test suite (hermetic — fake kitten, per-test tmp dirs; see docs/testing.md)
make test        # the whole hermetic suite (parallel by default, pytest-xdist)
make test-seq    # same, sequential (debugging / no xdist)
make test-all    # + the opt-in real-kitty smoke tests

# Lint (ruff, pinned in requirements-dev.txt; ruff.toml encodes docs/styleguide.md; CI-enforced)
make lint        # must stay clean
make lint-fix    # apply safe auto-fixes
```

To debug a reported session bug, prefer the **`audit-debug` skill** (`.claude/skills/audit-debug/SKILL.md`) — it walks the triage order (anomalies → errors → timeline → targeted SQL) and documents the full audit schema.

Script edits take effect immediately (hooks re-exec them). Only `kitty.conf` changes (e.g. `listen_on`) need a full kitty restart, and a renderer/interpreter change needs a mirror toggle off/on.

## Architecture

**Producer/renderer split via paint ops.** Hook handlers and tailers never print to the pane. They append width-INDEPENDENT **paint ops** (`rule`/`label`/`code`/`gut`/`line`, built by `core/ops.py`) as rows of the per-session state DB's `ops` table (`/tmp/claude-mirror-<session_id>.log.state.db` — the `.log` path is only the KEY everything derives from; no log file exists). The single renderer `claude-mirror.py`, running inside the pane, polls new rows by id, paints ops at the live pane width and re-renders everything on SIGWINCH so content reflows. Consequence: anything width-dependent (wrapping, gutters, dividers) belongs in the renderer; anything width-independent (syntax highlighting, one-liner reflow) runs once at op creation.

**Process model.** ~20 short-lived hook processes plus detached long-lived tailers/watchers, coordinating through SQLite: a per-session state DB (`core/state.py` — ops, scoreboard, slots, hand-offs; the live DB is in `/tmp` but at SessionEnd it is MOVED to the durable park `~/.claude/kitty-mirror-history/` — `core/paths.py` `HISTORY_DIR`/`parked_db` — and restored on resume, so a resumed session replays its mirror history even across a machine reboot that wipes `/tmp`), a global window-keyed tab DB (`/tmp/claude-kitty-tab.db` — tab colour + watcher pid locks), and `~/.claude/kitty-mirror.db` (remembered pane sizes). The only plain files left are what physics demands: the fg tee'd `.out` streams + `.done` sentinels (written by the rewritten command itself). Everything is Python; `claude-tab-status.py` reads the state/tab DBs read-only (`mode=ro`, so a probe can never create a DB whose existence is a liveness signal):
- `claude-cmd-pre.py` (PreToolUse Bash) rewrites foreground commands via `updatedInput` to tee output into a side file so it streams live; `claude-cmd-fmt.py` (PostToolUse) hands the real outcome to the tailer via a take-once hand-off record in the state DB. A **subagent's** foreground command is teed the same way (keyed by `tool_use_id`), but `claude-substream.py` spawns its tailer and hands off the outcome; gated by `CLAUDE_MIRROR_LIVE_FG_SUB`.
- `claude-stream.py` tails background/monitor/fg output files; `claude-substream.py` tails a subagent/teammate transcript (`subagents/agent-<id>.jsonl`) — the only in-order source of its prompt/messages/tools/result; `claude-codex-watch.py` (one per session) discovers every codex run from two global directories and spawns `claude-codex-stream.py` per run.
- Detached processes are spawned with `start_new_session=True` — spawning from a hook with bash `&` leaves them in the hook's process group, which Claude Code waits to drain (this hung SessionStart once; see `claude-codex-launch.py`).

**Slot rows** (`core/slots.py`, the state DB's `live` table) do double duty: they assign each concurrent stream a palette color (5-color palettes per kind: bg/monitor/subagent/teammate/codex) AND they are the tab tracker's liveness signal — `claude-tab-status.py stop` keeps the tab blue exactly while a live-pid row (kind `bg`/`monitor`/`fg`/`sub.pid`) exists. Rows store the owner pid and are liveness-checked (`core.state.pid_alive` — the ONE probe, where EPERM = exists-but-foreign-owned = alive) before being trusted; stale ones are stolen.

its block RENDERING lives in `substream_render.py` — an import-safe `Renderer` class with `_USE`/`_RESULT` per-tool-kind dispatch tables, identity + tailer hooks injected by the lifecycle module)/`tabstatus.py`/`split.py` (`stop_fmt.py` is the `Stop`/`StopFailure` producer that folds the final-turn tail into the scoreboard, the one sanctioned Stop-time state-DB writer — distinct from the tab dispatcher, which stays `mode=ro`). **Frontend (`frontends/`):** `base.py` (the `Frontend` interface + inert stub), `kitty.py` (kitten binary lookup, silenced `kitten @` calls, `iter_windows`, `set_tab_color`, `set_tab_title` (the web rename's sticky live-tab retitle), socket resolution, the `neighbors`/`groups` geometry walk).

**Everything is keyed by `session_id`** — the state DB, pane kitty vars (`claude_mirror`/`claude_session`), sidecars — so parallel sessions never collide. Exception: background-job detection is per-project (temp slug from cwd), so two sessions in one directory can cross-talk.

## Hard-won invariants (violating these reintroduces fixed bugs)

- **Hooks must never block or fail.** Every hook path exits 0 and swallows exceptions — but every swallow site must record to the audit first (the audit `errors` table). The tab-color path writes audit rows fire-and-forget.
- **Claude Code fires NO hook on cancel/interrupt** — no Stop, nothing. Every cancellation path needs its own recovery signal: writer-liveness for commands, `meta.json` `stoppedByUser` for a killed/cancelled subagent, the **parent transcript's `tool_result`** (keyed by the agent's `meta.json` `toolUseId`) for a *rejected/abandoned* Task — which fires neither `SubagentStop` nor `stoppedByUser` — a **`StopFailure` carrying the subagent's `agent_id`** for a subagent turn that *died on an API error* (e.g. 529 — fires no `SubagentStop`; `claude-stop-fmt.py` hands it to `subagent_fmt.finalize`, the same finaliser `SubagentStop` uses; async agents can't be recovered via the parent `tool_result` because its only entry is the "launched successfully" ack), and the transcript's `[Request interrupted by user]` line for plain replies (`interrupt-watch` — which must check what FOLLOWS the line: a queued message is delivered the instant the interrupt lands, its user-prompt record right after the line is the tell, and flipping green there painted "done" over the delivered turn's think). All are **events**, never idle timeouts: cancel-before-first-hook has no signal at all and is deliberately left unhandled for a TERMINAL Esc — do not re-add an idle-timeout backstop; it false-positived on every long think. The ONE sanctioned exception is the dashboard's `escape-recheck` (docs/tab-colors.md): a WEB interrupt is itself an event we generated, so its ~2s recheck may flip the dead magenta — but only on total silence (any tab-state movement or transcript growth over the press-time baseline bails).
- **Main session only:** any hook event carrying an `agent_id` is a subagent/teammate inner call — tab dispatch ignores it, and cmd/file formatters skip it (the substream owns subagent rendering; handling both would duplicate/mis-order). **Exception:** `claude-cmd-pre.py` does tee a subagent's foreground command for live streaming (marker + `updatedInput` only — the substream still owns rendering).
- **Release slot rows *before* calling `bg-recheck`**, or the recheck sees its own row. `bg-recheck` flips only a currently-blue tab and only when no live row remains.
- **Duplicate events are real:** `SubagentStart` and `SubagentStop` can each fire more than once for background agents — both handlers guard on slot state.
- **Failures arrive on `PostToolUseFailure`, not `PostToolUse`** — any new PostToolUse hook must be wired to both or failures silently vanish.
- **SessionStart can arrive with a SCRUBBED env** — Claude Code's agents view spawns `claude daemon run`, whose hook children have no `KITTY_WINDOW_ID`/`KITTY_LISTEN_ON` (the socket still resolves via the ppid walk, so pane calls *work*, anchorless). Pane ops must anchor to `KITTY_WINDOW_ID` or the `claude_session=<sid>`-tagged window and SKIP the whole lifecycle when neither exists (a daemon agent session / headless `claude -p` has no pane) — the focused-tab fallback let a phantom session close the focused tab's mirror and open an empty one (docs/mirror-pane.md, *Anchoring, and daemon-origin SessionStarts*).
- **`--resume` AND backgrounding a session can FORK the session id** — the conversation continues under a NEW sid with no SessionStart of its own (resume: SessionStart fires under the OLD sid, `source=resume`; backgrounding: the new sid is the background-job id), so the mirror/scorebar/tab freeze on the old sid. `plugins/claude_code/adopt.py` (run first by `dispatch.py`) adopts the predecessor at the fork's first event — renames its state DB to the new sid's path (symlinks at the old paths), retags the panes, writes the missing `sessions` row — driven by the take-once `adopt_pending` note (written by `split.cmd_open` for every HOSTED session) + `sids` registry in the global tab DB. The un-adopted regression is the canned anomaly "hook traffic under a sid with no sessions row" (docs/mirror-pane.md, *Resume (and backgrounding) can FORK the sid*).
- **Replayed output must not execute: every op's text passes `render.neutralize()` at paint** (only SGR styling and OSC 8 hyperlinks survive). The ops stream carries RAW command output and the renderer replays it on every reflow — an escape sequence in some command's output re-executes forever, not once (a tee'd `@kitty-cmd` DCS scrolled the mirror to the top on every repaint, masquerading as a scroll bug). Any new paint path must route op text through `_render`/neutralize.
- Empirically-confirmed but undocumented Claude Code behaviors this repo depends on (`updatedInput` command rewriting, Ctrl+B's `backgroundTaskId`+`backgroundedByUser` payload, `stoppedByUser` in meta.json, the interrupted-transcript line) are called out in the docs/ design docs (docs/streaming.md, docs/subagents.md, docs/tab-colors.md) — check there before assuming a payload field exists or not.

## Every new feature must be audit-covered

The audit trail is only useful if it has no blind spots — a mechanism that leaves no rows is undebuggable after the fact (the no-hook-on-cancel bug class was only cracked once auditing existed). When adding or changing a feature, wire it into the audit (`core/audit.py`) **in the same commit**:

- **New hook handler** → call `A.hook_event(payload, handler=, decision=)` with a `decision` string that says what the handler chose to do and why (the decision column is what makes `hook_events` diagnostic, not just a log).
- **New detached process / tailer** → `A.stream_start(...)` on spawn and `A.stream_end(stream_id, end_reason=)` on every exit path, plus `A.spawn(...)` at the launch site (wrap the tailer in `core.tail.stream_lifecycle` and spawn via `hookkit.spawn_streamer` and all of this comes for free). A stream with `ended_at IS NULL` is an anomaly signal — don't create streams that legitimately never end.
- **New coordination/marker/sentinel file** → `A.state_file(log, path, action, content)` on write and remove; new slot kinds go through `core/slots.py` so `A.slot(...)` rows come for free.
- **New tab-state input** → record via `A.transition(...)` with `applied` and a `reason`, including (especially) the bailed/skipped paths.
- **New swallow site** → `A.error(...)` before the `except: pass` (existing invariant above — it applies to new code too).
- **New accounting/derived numbers** (scoreboard fields, cost math) → make the inputs reconstructible: either the raw source is already audited (e.g. the transcript path in `sessions`) or add rows for it. "The number is wrong" must be answerable from the DB plus the named source, as the `message.id` token-dedup bug was.
- Then extend the **`anomalies` canned queries** in `core/audit.py` if the feature has a known failure signature, and update **`.claude/skills/audit-debug/SKILL.md` in BOTH places**: the **schema table** (what the new rows/columns/kinds are) *and* the **"Known bug shapes" playbook** (which symptom should now consult that evidence, and what the tell-tale value looks like). These drift independently — the schema table alone got updated once while the playbook kept pointing at the old sources, which makes the skill *look* current while it triages blind. A new evidence source that no bug shape references is a smell: either wire it into an existing shape or add the shape it exists for. New tables/columns belong in the same migration style as the existing ones (WAL-safe, spool-file fallback, never raise into a hook).

## Style rules live in docs/styleguide.md

**Read `docs/styleguide.md` before writing or reviewing code.** It is the normative companion to `docs/architecture.md`: layout and the hyphen/underscore naming rule, the dependency rule, the single-owner vocabulary table (which module owns which shared fact — re-encoding one is a bug, several are grep-test-enforced), import-time purity (no argv/I/O/frontend/DB work at import — the dispatcher imports every handler on every event), registries over if/elif ladders, audit-before-swallow, SQL binding + `mode=ro` probe rules, named constants for tuning/protocol literals, "deliberately different — don't unify" markers, byte-identical refactoring discipline, and the test conventions (`wait_until` only, hermetic tmpdirs, seed via product APIs, env knobs). New shared facts get an owner and a row in that table in the same commit.

## docs/ is the design doc

The `docs/` directory (indexed by `docs/README.md`) is the authoritative, exhaustively-detailed record of how every mechanism works *and why the alternatives failed*. When changing behavior, update the corresponding docs/ file in the same commit — the "why not X" notes there are what prevents regressing to already-rejected designs. The repo-root README.md is a short GitHub-facing overview only; internals belong in docs/.

## Branch in an isolated worktree, merge to main, push

This is a personal repo with a linear `main` history — there is no PR/review flow — but MULTIPLE agents often work here concurrently, and an agent that edits the shared checkout tramples (and gets confused by) the others' uncommitted changes. So:

1. **Never work directly in the main checkout.** Before touching any file, create a fresh branch off the latest `main` in an **isolated git worktree** (a branch alone does NOT isolate you — everyone in the same directory shares the same working tree and sees each other's uncommitted files). Use the harness's worktree isolation (`EnterWorktree`, or `isolation: "worktree"` for subagents) when available; otherwise `git worktree add <dir> -b <branch> main` yourself.
2. Do all work and commits on that branch, in that worktree.
3. When the change is complete and verified (code + its audit wiring + docs/ + skill docs, per the sections above, and `make test` green), **merge it into `main` and `git push`** — every time, without being asked. Keep history linear: rebase the branch onto the latest `main` first, then fast-forward merge (`git merge --ff-only`). If `main` moved underneath you, rebase and re-verify before merging.
4. Clean up: remove the worktree and delete the merged branch. Do not open PRs or leave work sitting on an unmerged branch.

(This is about *using* worktrees — the `WorktreeCreate`/`WorktreeRemove` hooks must still stay unwired, per the wiring section above.)
